3. Proposed database schema

Supabase Postgres should store authoritative room, player, property, event, and snapshot state.

Table Purpose Key fields
rooms Room, lobby, and game lifecycle. code, status, host_player_id, current_turn_player_id, winner_player_id, turn_number
players Players within rooms. room_id, display_name, avatar_key, is_host, is_ready, money, position, status
game_tiles Static tile definitions for the original board. tile_index, type, name, price, rent, amount, color_group
properties Room-specific property ownership. room_id, tile_id, owner_player_id, purchased_at
chance_cards Original chance card definitions. card_key, type, title, description, amount, steps
game_events Append-only visible event log. room_id, player_id, event_type, message, payload
game_snapshots Compact room state snapshots for hydration, recovery, and debugging. room_id, state, created_at

Add indexes for room lookup, room-scoped players/properties/events, and latest snapshots. Add constraints for six-character room codes, valid statuses, tile types, and card types.

4. Proposed app routes and API routes

Pages

  • src/app/page.tsx: landing page with create-room and join-room forms.
  • src/app/room/[roomCode]/page.tsx: room page that displays lobby, active game, or finished state.

API routes

  • src/app/api/rooms/create/route.ts: create room, generate unique six-character code, create host player.
  • src/app/api/rooms/join/route.ts: validate room and add player if lobby has fewer than four players.
  • src/app/api/game/start/route.ts: host-only start route for rooms with two to four players.
  • src/app/api/game/roll-dice/route.ts: current-turn-only server dice roll and tile resolution.
  • src/app/api/game/buy-property/route.ts: current-player purchase validation and ownership update.
  • src/app/api/game/end-turn/route.ts: advance to next non-bankrupt player.

For the MVP, local storage can hold playerId after room creation or joining. Each API route must validate that the submitted player ID belongs to the requested room.

6. Proposed game engine structure

src/lib/game/types.ts
src/lib/game/board.ts
src/lib/game/cards.ts
src/lib/game/rules.ts
src/lib/game/engine.ts

Module responsibilities

  • types.ts: shared game types for rooms, players, tiles, cards, properties, dice, events, and game state.
  • board.ts: original 24-tile MVP board definition.
  • cards.ts: original chance card definitions.
  • rules.ts: pure helpers for dice, movement, rent, tile effects, bankruptcy, winner detection, and next-turn selection.
  • engine.ts: orchestrates startGame, rollDice, buyProperty, and endTurn.

Recommended original tile names include cozy pixel-world concepts such as Mochi Market, Pixel Pier, Cloud Bakery, Lantern Lane, Bamboo Arcade, and Sunny Studio.

7. Supabase Realtime strategy

  1. Initial page load fetches the current room state by room code.
  2. The client subscribes to room-scoped changes for rooms, players, properties, and game_events.
  3. When realtime events arrive, update local state incrementally or refetch compact room state.
  4. Use game_events for the visible event log.
  5. Use game_snapshots for faster reload/reconnect recovery.
Safety rule: Realtime is display-only. It must never authorize or compute game actions.

8. Pixel art UI strategy using Tailwind/CSS first

Build a cute retro pixel board game style without external image assets.

  • Use CSS and Tailwind utility classes for pixel borders, shadows, dice, tokens, and cards.
  • Use hard-edged shapes, cozy colors, and image-rendering: pixelated.
  • Create a responsive original board layout instead of copying Monopoly board layout or color ordering.
  • Use original CSS-only player tokens such as pixel blobs, tiny robots, stars, or cozy abstract mascots.

10. Risks and how to avoid them

Risk Mitigation
Accidentally copying Monopoly-protected content Use original names, card text, tile layout, visual style, and token concepts. Do not use Monopoly branding, property names, artwork, board order, or card wording.
Client-side cheating Generate dice and calculate movement, money, rent, bankruptcy, and winner status only on the server through API routes.
Race conditions during turns Validate current turn at read and write time. Use safe update patterns or Supabase RPC transactions for critical mutations.
Room code collisions Use a unique constraint and retry six-character uppercase alphanumeric code generation on conflict.
Supabase key misuse Use environment variables only, keep .env.local out of git, and avoid privileged direct writes from client components.
