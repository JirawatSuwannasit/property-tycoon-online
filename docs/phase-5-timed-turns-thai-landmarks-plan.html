Phase 5 Plan: Timed Turns & Thai Landmark Properties

Updated implementation plan for Property Tycoon Online before coding the game engine. This phase keeps the server authoritative, adds 30-second turn decisions, and prepares the MVP board for original Thailand landmark/travel-inspired property rights.

30-second roll deadline Server auto-roll 30-second buy/upgrade decision Thai landmark theme No Monopoly assets

1. Phase 5 objective

Implement the server-side game engine foundation for dice rolls, movement, property ownership, rent, upgrades, event logging, and turn advancement while adding deadline-driven automatic actions. The browser may display countdowns, but it must never decide the true timeout, dice result, money changes, rent, upgrades, bankruptcy, or winner state.

Important: Phase 5 should not build the full board UI, animations, trading, auctions, mortgage, houses, hotels, chat, bots, login, or advanced rules.

2. Product decisions to apply

  • Each player has 30 seconds to roll dice after their turn starts.
  • If the roll deadline expires, the server automatically rolls for that player.
  • After landing on a buyable or upgradeable property, the player has 30 seconds to decide.
  • If the buy/upgrade deadline expires, the server automatically skips the decision and advances the turn.
  • Property tiles should use famous landmarks or travel destinations in Thailand.
  • Avoid language that says players build houses directly on sacred or historical landmarks.
  • Use wording such as buy landmark rights, upgrade landmark, develop tourism zone, and upgrade visitor facilities.
  • Keep all names, rules text, board layout, art direction, and UI copy original. Do not copy Monopoly branding, names, board layout, card text, artwork, or protected assets.

4. Timed turn state machine

Phase Meaning Deadline behavior
waiting_to_roll Current player may roll dice. If expired, server auto-rolls and resolves movement.
resolving_roll Server is calculating dice, movement, tile effects, rent, bankruptcy, and pending decisions. No client action should be accepted.
waiting_to_buy_or_upgrade Current player is deciding whether to buy landmark rights or upgrade visitor facilities. If expired, server skips the decision and advances to the next active player.
turn_ending Server is advancing to the next active player and writing events/snapshots. No client action should be accepted.
finished Game has a winner. No turn deadlines remain.

5. Timeout rules

  1. When a player turn starts, set turn_phase = 'waiting_to_roll' and action_deadline_at = now() + interval '30 seconds'.
  2. If the deadline expires while waiting to roll, the server auto-rolls, moves the player, and resolves the landing tile.
  3. If the player lands on an unowned property, set turn_phase = 'waiting_to_buy_or_upgrade', pending_action = 'buy_property', pending_tile_id, and a new 30-second deadline.
  4. If the player lands on their own property and upgrade level is below 3, set pending_action = 'upgrade_property', pending_tile_id, and a new 30-second deadline.
  5. If the buy/upgrade deadline expires, the server clears pending fields, logs a skipped decision, and advances to the next active player.
  6. If a tile has no pending decision, the server resolves it and advances the turn immediately.

6. Server helper plan: resolveExpiredTurnAction(roomId)

Create a server helper in src/lib/server/turn-timeouts.ts or inside the game service layer.

export async function resolveExpiredTurnAction(roomId: string) {
  // 1. Load room and current state with admin client.
  // 2. If no action_deadline_at or deadline is in the future, return unchanged.
  // 3. Use a safe conditional update or transaction/RPC lock to avoid duplicate execution.
  // 4. If turn_phase is waiting_to_roll, auto-roll through the same engine path as manual roll.
  // 5. If turn_phase is waiting_to_buy_or_upgrade, skip pending action and advance turn.
  // 6. Write game_events and game_snapshots.
  // 7. Return updated game/lobby state.
}

Call sites

  • Before returning game/room state from server API routes.
  • Before processing roll, buy, upgrade, or end-turn actions.
  • From an optional scheduled endpoint or cron route in later deployment hardening.
The helper must be idempotent and safe against duplicate execution. Two clients may ask for state at the same time after a deadline has expired.

8. MVP property upgrade rules

  • Upgrade level starts at 0.
  • Maximum upgrade level is 3.
  • Upgrade is available only when the current player lands on their own property.
  • Level 1 upgrade cost = 50% of tile price.
  • Level 2 upgrade cost = 75% of tile price.
  • Level 3 upgrade cost = 100% of tile price.
  • Final rent = base_rent * (1 + upgrade_level).
  • Use player-facing copy such as “upgrade visitor facilities” or “develop tourism zone,” not “build houses on landmarks.”

10. API route plan for Phase 5

  • POST /api/game/start: initialize active game, first turn, deadline, and event log.
  • POST /api/game/roll-dice: validate current player and phase, call timeout resolver, then roll on the server.
  • POST /api/game/buy-property: validate pending buy action, current player, deadline, money, and ownership.
  • POST /api/game/upgrade-property: validate pending upgrade action, ownership, max level, cost, and deadline.
  • POST /api/game/skip-pending-action: optional explicit skip endpoint using the same server path as timeout auto-skip.
  • GET /api/game/[roomCode]: call resolveExpiredTurnAction before returning state.
