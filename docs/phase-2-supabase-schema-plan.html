3. Schema details

The schema should use server-authoritative state and enforce useful constraints without implementing gameplay yet.

Shared SQL setup

create extension if not exists pgcrypto;

create or replace function public.set_updated_at()
returns trigger
language plpgsql
as $$
begin
  new.updated_at = now();
  return new;
end;
$$;

rooms

  • id uuid primary key default gen_random_uuid()
  • room_code text unique with ^[A-Z0-9]{6}$ check.
  • status: waiting, playing, finished.
  • max_players integer default 4 constrained to 4.
  • Nullable host_player_id, current_turn_player_id, and winner_player_id.
  • turn_number integer default 0, created_at, and updated_at.

players

  • References rooms(id) with cascade delete.
  • seat_no constrained to 1 through 4 and unique per room.
  • status: active, bankrupt, left.
  • money integer default 1500, position integer default 0, jail_turns integer default 0.
  • session_token_hash nullable for future identity validation.
  • Add indexes on room_id and (room_id, status).

game_tiles

  • tile_index unique and constrained from 0 to 23.
  • type: start, property, tax, chance, jail, go_to_jail, bonus.
  • Property rows require price and rent.
  • amount supports future start, tax, and bonus effects.
  • Add indexes on type and color_group.

properties

  • Room-specific ownership rows.
  • References rooms(id), game_tiles(id), and nullable players(id).
  • Unique (room_id, tile_id).
  • Check that owner_player_id and purchased_at are both null or both present.

chance_cards

  • card_key unique and stable.
  • type: receive_money, pay_money, move_to_start, move_steps, go_to_jail.
  • Nullable amount, steps, and target_tile_index.
  • Constrain target_tile_index to 0 through 23 when present.

game_events

  • Append-only room event log for future display/refetch.
  • References rooms(id) and nullable players(id).
  • payload jsonb not null default '{}'::jsonb.
  • Index (room_id, created_at desc).

game_snapshots

  • Room state snapshots for future reconnect/refetch/debugging.
  • state jsonb constrained to a JSON object.
  • version integer default 1 with unique (room_id, version).
  • Index (room_id, created_at desc).

5. TypeScript type file locations

Phase 2 should add types that match the SQL schema but should not add API behavior or game logic.

src/lib/database.types.ts
src/lib/game/types.ts
src/lib/game/schema.ts

src/lib/database.types.ts

  • Define Json.
  • Define schema unions: RoomStatus, PlayerStatus, TileType, and ChanceCardType.
  • Define a Supabase-style Database type with Row, Insert, and Update shapes for all seven tables.
  • Later, Supabase helpers can be typed as createBrowserClient<Database>(...) and createServerClient<Database>(...).

src/lib/game/types.ts

  • Define game-facing type aliases and DTO-like shapes.
  • Examples: BoardTile, ChanceCard, RoomStatus, PlayerStatus, TileType, and ChanceCardType.

src/lib/game/schema.ts

  • Define schema-level constants only.
  • MVP_BOARD_SIZE = 24
  • MIN_PLAYERS_TO_START = 2
  • MAX_PLAYERS_PER_ROOM = 4
  • STARTING_MONEY = 1500
  • START_TILE_INDEX = 0
  • JAIL_TILE_INDEX = 8

6. Phase 2 boundaries

In scope

  • Supabase SQL migrations.
  • Seven required tables.
  • Constraints, indexes, defaults, and triggers.
  • Exactly 24 original board tiles.
  • About 10-12 original chance cards.
  • TypeScript schema/game types.

Out of scope

  • No room creation API.
  • No lobby logic.
  • No gameplay engine.
  • No dice/money/rent calculations.
  • No realtime subscriptions.
  • No unrelated UI work.

7. Risks before implementation

Risk Mitigation
Circular foreign keys between rooms and players. Create rooms first, then players, then add nullable player foreign keys back to rooms.
Same-room relationships are hard to fully enforce with simple FKs. Use normal FKs in Phase 2 and enforce same-room legality in later API/server logic. Consider triggers in a future hardening phase.
Seed data can drift if duplicated in TypeScript. Keep SQL seed data canonical in Phase 2. Add only types and schema constants in TypeScript.
Protected-content similarity. Use only original cozy pixel-world tile/card names and avoid Monopoly board layout, names, card wording, artwork, branding, and token designs.
Over-scoping Phase 2. Limit changes to migrations and types. Do not implement API routes, lobby behavior, realtime behavior, or game rules.
Realtime being mistaken for game authority. Do not add realtime logic in Phase 2. Later realtime must be display/refetch only.

8. First recommended Phase 2 implementation step

Start with the schema migration:

supabase/migrations/20260611000100_create_mvp_schema.sql
  1. Enable pgcrypto.
  2. Add set_updated_at().
  3. Create rooms.
  4. Create players.
  5. Add player foreign keys back to rooms.
  6. Create game_tiles, properties, chance_cards, game_events, and game_snapshots.
  7. Add indexes, constraints, and update triggers.
After the schema migration is in place, add the seed migration with the exact 24 board tiles and 12 chance cards, then add the TypeScript type files.
